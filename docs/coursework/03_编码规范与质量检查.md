# 第 3 项：编码规范与质量检查

## 1. 目标

建立统一的编码规范和质量检查规则，保证项目代码结构清晰、命名一致、注释规范、格式统一，并通过代码检查工具及时发现潜在缺陷。

编码规范主要覆盖以下内容：

- 命名规范；
- 目录与分层规范；
- Java 编码规范；
- HTML、CSS、JavaScript 编码规范；
- SQL 与 MyBatis 编码规范；
- 注释规范；
- 异常处理规范；
- 测试规范；
- 代码质量检查规范。

## 2. 目录与分层规范

项目应采用清晰的分层结构，不同类型代码放入对应目录，避免业务逻辑、数据库访问和页面代码混杂。

推荐目录结构如下：

```text
src
├── main
│   ├── java
│   │   └── com
│   │       └── zbw
│   │           ├── controller
│   │           ├── domain
│   │           ├── mapper
│   │           ├── service
│   │           │   └── impl
│   │           └── utils
│   └── resources
│       ├── db
│       ├── mapper
│       ├── static
│       ├── templates
│       └── application.yml
└── test
    └── java
```

各层职责如下：

| 层次 | 目录 | 职责 |
|---|---|---|
| 控制层 | `controller` | 接收请求、参数校验、调用业务层、返回结果 |
| 实体层 | `domain` | 定义与数据库表或业务对象对应的数据结构 |
| 数据访问层 | `mapper` | 定义数据库访问接口 |
| SQL 映射层 | `resources/mapper` | 编写 MyBatis XML SQL 映射 |
| 业务接口层 | `service` | 定义业务功能接口 |
| 业务实现层 | `service/impl` | 实现核心业务逻辑 |
| 工具层 | `utils` | 存放分页、日期、字符串、校验等工具类 |
| 页面模板 | `templates` | 存放 Thymeleaf 页面 |
| 静态资源 | `static` | 存放 CSS、JavaScript、图片和第三方前端库 |
| 数据库脚本 | `db` | 存放建表脚本和初始化数据 |
| 测试代码 | `test` | 存放单元测试和集成测试 |

分层原则：

1. Controller 不直接访问数据库；
2. Service 不直接拼接页面内容；
3. Mapper 不编写业务规则；
4. 页面 JavaScript 不承担后端权限判断；
5. 工具类只放通用逻辑，不放具体业务流程。

## 3. Java 命名规范

### 3.1 包命名

包名统一使用小写字母，避免使用下划线和大写字母。

示例：

```text
com.zbw.domain
com.zbw.mapper
com.zbw.service
com.zbw.service.impl
com.zbw.utils.page
```

### 3.2 类命名

类名使用大驼峰命名法。

| 类型 | 命名规则 | 示例 |
|---|---|---|
| 实体类 | 名词 | `User`、`Book`、`Department` |
| 查询条件类 | 实体名 + `Example` | `UserExample`、`BookExample` |
| Mapper 接口 | 实体名 + `Mapper` | `UserMapper`、`BookMapper` |
| Service 接口 | `I` + 业务名 + `Service` | `IUserService`、`IBookService` |
| Service 实现类 | 业务名 + `ServiceImpl` | `UserServiceImpl` |
| VO 类 | 业务名 + `Vo` | `BookVo`、`BorrowingBooksVo` |
| 工具类 | 功能名 + `Util` 或明确名词 | `Page`、`DateUtil` |

### 3.3 方法命名

方法名使用小驼峰命名法，名称应表达具体动作。

推荐：

```java
findUserByUserName()
userLogin()
updateUser()
findBooksByCategoryId()
selectBooksByBookPartInfo()
deleteUserById()
```

不推荐：

```java
doIt()
test()
handle()
getData()
aaa()
```

方法命名建议：

- 查询单个对象使用 `findBy...` 或 `selectBy...`；
- 查询列表使用 `findAll...` 或 `selectAll...`；
- 新增数据使用 `insert...` 或 `add...`；
- 修改数据使用 `update...`；
- 删除数据使用 `delete...`；
- 登录校验使用 `login` 或 `...Login`；
- 布尔判断方法使用 `is...`、`has...`、`can...`。

### 3.4 变量命名

变量名使用小驼峰命名法，避免无意义缩写。

推荐：

```java
userName
bookId
pageNum
pageSize
borrowingBooksList
bookCategoryId
```

不推荐：

```java
u
b
p
list1
data
temp
```

循环变量可以使用简短名称，但复杂逻辑中应使用有意义名称。

## 4. Java 格式规范

### 4.1 缩进与空格

Java 代码统一使用 4 个空格缩进，不使用 Tab 混排。

示例：

```java
public boolean userBorrowingBook(int bookId, HttpServletRequest request) {
    User user = (User) request.getSession().getAttribute("user");
    if (user == null) {
        return false;
    }
    return true;
}
```

### 4.2 大括号规范

类、方法、条件语句、循环语句的大括号与声明同行。

推荐：

```java
if (user == null) {
    return false;
}
```

不推荐：

```java
if (user == null)
{
    return false;
}
```

### 4.3 空行规范

以下位置应使用空行分隔：

- 成员变量与方法之间；
- 不同业务逻辑代码块之间；
- return 前的复杂处理逻辑之后；
- import 分组之间。

不应使用过多连续空行。

### 4.4 方法长度

单个方法应保持职责单一。

建议：

- 普通方法不超过 80 行；
- 复杂业务方法应拆分为多个私有方法；
- 重复逻辑应提取为工具方法或公共业务方法。

## 5. 注释规范

### 5.1 类注释

核心类应说明类的用途。

示例：

```java
/**
 * 用户业务逻辑实现类，负责用户登录、借书、还书和用户信息维护。
 */
@Service
public class UserServiceImpl implements IUserService {
}
```

### 5.2 方法注释

公共方法、接口方法和复杂业务方法应添加方法注释。

示例：

```java
/**
 * 根据图书分类分页查询图书列表。
 *
 * @param categoryId 图书分类 ID
 * @param pageNum 当前页码
 * @return 分页后的图书展示对象
 */
Page<BookVo> findBooksByCategoryId(int categoryId, int pageNum);
```

### 5.3 行内注释

行内注释只用于解释不直观的业务规则。

推荐：

```java
calendar.add(Calendar.MONTH, 2); // 借阅期限为两个月
```

不推荐：

```java
int n = 0; // 定义一个 int 类型变量 n
```

### 5.4 编码要求

所有注释必须使用 UTF-8 编码，禁止提交乱码注释。

如果发现乱码，应重新设置文件编码并修复注释内容。

## 6. 异常处理规范

### 6.1 基本原则

异常处理应遵循以下原则：

1. 不吞掉异常；
2. 不直接向页面暴露底层异常信息；
3. 数据库写操作失败时应记录日志；
4. 业务异常和系统异常应区分处理；
5. Controller 层应返回统一错误信息。

### 6.2 不推荐写法

```java
try {
    mapper.insert(record);
} catch (Exception e) {
    return false;
}
```

问题：

- 没有记录异常原因；
- 调用方无法判断失败类型；
- 排查问题困难。

### 6.3 推荐写法

```java
try {
    int result = mapper.insert(record);
    return result > 0;
} catch (Exception e) {
    log.error("新增借阅记录失败，bookId={}, userId={}", bookId, userId, e);
    return false;
}
```

如果项目中定义了统一异常类，也可以抛出业务异常：

```java
throw new BusinessException("借阅失败，请稍后重试");
```

## 7. 事务规范

涉及多个数据库写操作的业务必须使用事务，避免部分成功、部分失败。

适合添加事务的场景：

- 借书；
- 还书；
- 删除用户及其相关借阅记录；
- 删除图书及其相关借阅记录；
- 批量导入数据；
- 批量删除数据。

示例：

```java
@Transactional
public boolean userBorrowingBook(int bookId, HttpServletRequest request) {
    // 检查图书是否可借
    // 新增借阅记录
    // 返回处理结果
}
```

## 8. MyBatis 与 SQL 规范

### 8.1 Mapper 接口规范

Mapper 接口方法名应与 XML 中的 SQL ID 保持一致。

示例：

```java
List<Book> selectByCategoryId(@Param("categoryId") int categoryId,
                              @Param("currIndex") int currIndex,
                              @Param("pageSize") int pageSize);
```

对应 XML：

```xml
<select id="selectByCategoryId" resultMap="BaseResultMap">
    select *
    from book
    where book_category = #{categoryId}
    limit #{currIndex}, #{pageSize}
</select>
```

### 8.2 参数绑定规范

SQL 参数应优先使用 `#{}`，避免 SQL 注入风险。

推荐：

```xml
where book_name like #{bookPartInfo}
```

谨慎使用：

```xml
order by ${orderByClause}
```

`${}` 会直接拼接 SQL，必须确保参数来源安全。

### 8.3 SQL 格式规范

SQL 语句应保持缩进清晰。

推荐：

```xml
<select id="selectAllByPage" resultMap="BaseResultMap">
    select *
    from borrowingbooks
    limit #{currIndex}, #{pageSize}
</select>
```

### 8.4 数据表字段规范

数据库字段使用小写字母和下划线。

示例：

```text
user_id
user_name
book_id
book_name
book_category
```

Java 实体属性使用小驼峰：

```text
userId
userName
bookId
bookName
bookCategory
```

字段映射通过 MyBatis `resultMap` 完成。

## 9. 前端编码规范

### 9.1 HTML 规范

HTML 页面应保持结构清晰。

要求：

- 使用 UTF-8 编码；
- 页面标题应准确描述页面功能；
- 公共头部、底部应抽取为公共模板；
- 表单字段应设置明确的 `name` 和 `id`；
- 避免在 HTML 中堆积大量 JavaScript 逻辑。

### 9.2 CSS 规范

CSS 应放在独立样式文件中，避免大量行内样式。

命名建议：

```css
.login-form
.book-list
.user-info
.admin-header
```

### 9.3 JavaScript 规范

JavaScript 函数命名应表达具体动作。

推荐：

```javascript
findAllBookCategory()
submitUserInfo()
deleteBookById()
showBorrowingRecord()
```

不推荐：

```javascript
fun1()
test()
abc()
doSomething()
```

Ajax 请求应统一处理成功和失败情况。

示例：

```javascript
$.ajax({
    type: "post",
    url: "/findAllBookCategory",
    dataType: "json",
    success: function (data) {
        // 渲染分类列表
    },
    error: function () {
        layer.msg("请求失败，请稍后重试");
    }
});
```

## 10. 测试规范

### 10.1 测试范围

测试应覆盖以下内容：

- Mapper 数据查询；
- Mapper 新增、修改、删除；
- Service 业务逻辑；
- 登录成功和登录失败；
- 借书成功和重复借书；
- 还书成功和还书失败；
- 分页边界；
- 空结果处理；
- Controller 参数校验。

### 10.2 测试命名

测试类命名：

```text
UserServiceTest
BookMapperTest
BorrowingBooksMapperTest
```

测试方法命名：

```java
testUserLoginSuccess()
testFindBooksByCategoryId()
testBorrowBookWhenBookAlreadyBorrowed()
```

### 10.3 断言规范

测试中应使用断言判断结果，不应只打印控制台。

推荐：

```java
assertNotNull(user);
assertEquals("user1", user.getUserName());
assertTrue(result);
```

不推荐：

```java
System.out.println(user);
```

控制台输出可以辅助调试，但不能替代断言。

## 11. 代码质量检查工具

项目可使用 SonarLint 进行本地代码质量检查。

### 11.1 SonarLint 检查内容

SonarLint 可用于检查：

- 空指针风险；
- 重复代码；
- 无用变量；
- 未使用 import；
- 过长方法；
- 复杂条件判断；
- 未处理异常；
- 潜在 SQL 注入风险；
- 资源未关闭；
- 命名不规范。

### 11.2 使用要求

开发人员在提交代码前应执行以下检查：

1. IDE 无明显编译错误；
2. SonarLint 无严重问题；
3. 新增代码符合命名规范；
4. 新增 SQL 使用安全参数绑定；
5. 新增业务逻辑有必要测试；
6. 没有提交乱码、临时输出和无意义注释。

## 12. 代码提交前检查清单

提交代码前应检查以下事项：

| 检查项 | 要求 |
|---|---|
| 编码 | 文件统一 UTF-8 |
| 命名 | 类名、方法名、变量名符合规范 |
| 分层 | Controller、Service、Mapper 职责清晰 |
| 注释 | 复杂业务逻辑有必要注释 |
| 异常 | 不吞掉异常，必要时记录日志 |
| SQL | 使用 `#{}` 绑定参数 |
| 事务 | 多步写操作使用事务 |
| 测试 | 新增或修改功能有对应测试 |
| 前端 | Ajax 有失败处理 |
| 配置 | 不提交个人密码和本地绝对路径 |
