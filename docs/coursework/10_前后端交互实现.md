# 第10项 前后端交互实现

## 1. 实践目标

前后端交互实现的目标是将页面操作与后端业务处理连接起来，使用户在浏览器中完成登录、查询、借书、还书、资料修改等操作时，系统能够把请求参数提交到后端，由后端完成业务判断和数据处理，再把处理结果返回给前端页面。

本项工作重点包括：

1. 编写前端请求代码，完成页面事件与后端接口之间的绑定。
2. 使用异步请求实现不刷新整页的数据交互。
3. 在请求前进行必要的参数校验，降低无效请求进入后端的概率。
4. 对请求成功、请求失败、业务失败分别给出页面反馈。
5. 补充统一 AJAX 辅助代码，为加载状态和异常处理提供公共封装。

## 2. 技术路线

系统采用服务端页面渲染与前端异步请求结合的方式实现交互：

| 层次 | 使用技术 | 作用 |
| --- | --- | --- |
| 页面模板 | Thymeleaf | 渲染登录页、用户页、管理员页和表格数据 |
| 前端脚本 | JavaScript、jQuery | 监听按钮点击、表单提交、分页跳转等事件 |
| 弹窗反馈 | Layui layer | 显示成功、失败、确认、提示信息 |
| 请求方式 | 表单提交、jQuery AJAX | 完成页面跳转式请求和异步数据请求 |
| 后端接口 | Spring MVC 风格接口约定 | 接收前端参数，调用业务层处理数据 |
| 数据格式 | 表单字段、JSON、布尔值、页面模型数据 | 在前端与后端之间传递输入和处理结果 |

整体流程如下：

```text
页面事件触发
  -> 前端读取表单字段或按钮参数
  -> 前端进行空值、数字格式、页码范围等基础校验
  -> 通过表单提交或 AJAX 请求访问后端接口
  -> 后端接口接收参数并调用业务逻辑层
  -> 业务逻辑层调用数据访问层完成数据库操作
  -> 后端返回页面、JSON 数据或布尔处理结果
  -> 前端根据返回值显示提示、刷新页面或更新局部数据
```

## 3. 交互方式划分

系统中的交互分为两类。

第一类是页面跳转式交互。该方式适用于登录、菜单跳转、分页查询等场景。浏览器提交表单后由后端返回新的页面或带模型数据的页面。

第二类是 AJAX 异步交互。该方式适用于新增图书、维护分类、借阅图书、归还图书、修改个人信息等场景。页面不需要整体跳转，只需要将处理结果提示给用户，必要时刷新当前页面或重新加载下拉框数据。

## 4. 页面跳转式请求

页面跳转式请求主要用于用户登录、管理员登录、图书查询和列表筛选。它的特点是请求结果通常会改变当前页面。

| 页面功能 | 请求目标 | 参数来源 | 后端处理职责 | 前端结果 |
| --- | --- | --- | --- | --- |
| 用户登录 | `/userLogin` | 用户名、密码 | 校验读者账号，写入会话状态 | 登录成功进入用户首页，失败返回提示 |
| 管理员登录 | `/adminLogin` | 管理员账号、密码 | 校验管理员账号，写入会话状态 | 登录成功进入管理首页，失败返回提示 |
| 图书搜索 | `/findBookByBookPartInfo` | 图书名称、作者、出版社等筛选条件 | 按条件查询图书 | 返回查询结果页 |
| 管理员图书筛选 | `/showBooksResultPageByCategoryId` | 图书分类编号、页码 | 按分类分页查询图书 | 返回管理员图书列表页 |

页面跳转式交互适合需要重新渲染表格或进入新页面的操作。它不依赖 AJAX 回调，而是由后端重新组织页面模型数据。

## 5. AJAX 异步请求清单

AJAX 异步请求集中在管理员端和用户端脚本中。前端通过 `$.ajax` 向接口提交参数，后端返回 JSON 数据、布尔值或操作结果。

| 序号 | 前端脚本 | 接口 | 请求参数 | 返回结果 | 页面行为 |
| --- | --- | --- | --- | --- | --- |
| 1 | `scripts/admin/addBook.js` | `/addBook` | 图书表单字段 | 操作结果 | 添加成功后提示并刷新页面 |
| 2 | `scripts/admin/addBook.js` | `/findAllBookCategory` | 无 | 图书分类数组 | 填充图书分类下拉框 |
| 3 | `scripts/admin/addBookCategory.js` | `/addBookCategory` | 分类名称等字段 | 操作结果 | 添加成功后提示并刷新页面 |
| 4 | `scripts/admin/addBookCategory.js` | `/deleteCategory` | 分类编号 | 操作结果 | 删除成功后提示并刷新页面 |
| 5 | `scripts/admin/addUser.js` | `/addUser` | 用户表单字段 | 操作结果 | 添加成功后提示并刷新页面 |
| 6 | `scripts/admin/adminInfo.js` | `/updateAdmin` | 管理员资料字段 | 布尔值或操作结果 | 修改成功后关闭弹层并刷新 |
| 7 | `scripts/admin/showBooks.js` | `/findAllBookCategory` | 无 | 图书分类数组 | 为筛选框填充分类数据 |
| 8 | `scripts/admin/showUsers.js` | `/deleteUser` | 用户编号 | 操作结果 | 删除成功后提示并刷新 |
| 9 | `scripts/user/borrowingBook.js` | `/userBorrowingBook` | 图书编号 | 布尔值 | 根据返回值提示借书成功或失败 |
| 10 | `scripts/user/returnBook.js` | `/userReturnBook` | 图书编号 | 布尔值 | 根据返回值提示还书成功或失败 |
| 11 | `scripts/user/userMessage.js` | `/updateUser` | 用户资料字段 | 布尔值或操作结果 | 修改成功后关闭弹层并刷新 |

这些接口覆盖了系统的主要交互动作：数据查询、数据新增、数据删除、资料修改、借阅处理、归还处理。

## 6. 前端脚本职责分析

### 6.1 管理员新增图书交互

文件：`scripts/admin/addBook.js`

该脚本负责管理员新增图书页面的交互处理。

核心逻辑包括：

1. 使用 Layui 的 `form.on('submit(btn1)', ...)` 监听提交按钮。
2. 调用 `addBook()` 函数提交图书表单。
3. 通过 `$('#addBookForm').serialize()` 将表单字段序列化。
4. 使用 POST 请求访问 `/addBook`。
5. 请求成功后调用 `layer.msg` 显示成功提示，并在短暂延迟后刷新页面。
6. 页面加载完成后调用 `findAllBookCategory()`，向 `/findAllBookCategory` 请求分类数据。
7. 将返回的分类数组转换成 `<option>` 元素，填入图书分类下拉框。

该交互保证管理员进入新增图书页面时可以直接选择已有分类，并且新增图书后能看到最新页面状态。

关键请求结构：

```javascript
$.ajax({
    async: false,
    type: 'post',
    url: '/addBook',
    data: $('#addBookForm').serialize(),
    success: function (data) {
        layer.msg('add success', {time: 1500, icon: 1});
        setTimeout(function () {
            location.reload();
        }, 1500);
    },
    error: function (data) {
        alert('add failed');
    }
});
```

### 6.2 管理员维护图书分类交互

文件：`scripts/admin/addBookCategory.js`

该脚本负责图书分类的新增、删除和分页边界控制。

核心逻辑包括：

1. 监听分类新增表单提交事件。
2. 将分类表单字段序列化后提交给 `/addBookCategory`。
3. 删除分类前使用确认弹窗，避免误删。
4. 确认删除后向 `/deleteCategory` 提交分类编号。
5. 对上一页、下一页按钮进行页码边界判断，防止跳转到无效页码。

该交互把危险操作和普通操作区分开：新增分类直接提交，删除分类先确认再请求。

### 6.3 管理员新增用户交互

文件：`scripts/admin/addUser.js`

该脚本负责管理员创建读者账号。

核心逻辑包括：

1. 监听新增用户表单提交。
2. 使用 `$('#addUserForm').serialize()` 获取用户编号、姓名、密码、院系等字段。
3. 通过 POST 请求访问 `/addUser`。
4. 请求成功后显示成功提示并刷新页面。
5. 请求失败时给出失败提示。

该交互将用户创建过程保持在当前页面完成，适合管理员连续录入账号。

### 6.4 管理员个人信息修改交互

文件：`scripts/admin/adminInfo.js`

该脚本负责管理员修改个人资料。

核心逻辑包括：

1. 点击修改按钮后打开 Layui 弹层。
2. 在弹层内展示管理员资料表单。
3. 提交时序列化表单数据。
4. 向 `/updateAdmin` 提交修改请求。
5. 根据后端返回结果提示成功或失败。
6. 成功后关闭弹层，并刷新页面显示最新资料。

该交互属于典型的局部编辑场景。弹层减少页面跳转，使资料修改过程更直接。

### 6.5 管理员图书列表筛选交互

文件：`scripts/admin/showBooks.js`

该脚本负责图书管理列表页的分类筛选和分页辅助。

核心逻辑包括：

1. 页面加载时请求 `/findAllBookCategory`。
2. 将分类结果写入筛选下拉框。
3. 管理员选择分类后，通过表单请求后端筛选图书。
4. 分页按钮点击前检查页码是否越界。
5. 当已经是第一页或最后一页时，给出提示而不是继续跳转。

该交互避免无效分页请求，同时使分类筛选项和数据库分类数据保持一致。

### 6.6 管理员用户列表删除交互

文件：`scripts/admin/showUsers.js`

该脚本负责读者用户列表中的删除操作和分页控制。

核心逻辑包括：

1. 点击删除按钮后读取用户编号。
2. 使用确认弹窗二次确认。
3. 确认后向 `/deleteUser` 提交删除请求。
4. 删除成功后提示并刷新列表。

### 6.7 用户借书交互

文件：`scripts/user/borrowingBook.js`

该脚本负责读者借阅图书。

核心逻辑包括：

1. 监听借书按钮点击事件。
2. 读取输入框中的图书编号。
3. 对图书编号进行空值和数字格式校验。
4. 校验不通过时直接在页面提示，不发送请求。
5. 校验通过后向 `/userBorrowingBook` 提交图书编号。
6. 后端返回 `true` 时提示借书成功。
7. 后端返回 `false` 时提示借书失败。

该交互把“输入合法性”和“业务是否允许借书”分开处理。前端负责基础格式检查，后端负责库存、重复借阅、用户状态等业务判断。

关键请求结构：

```javascript
$.ajax({
    async: false,
    type: 'post',
    url: '/userBorrowingBook',
    dataType: 'json',
    data: {bookId: bookId},
    success: function (data) {
        if (data.toString() == 'true') {
            layer.msg('borrow success', {icon: 6, time: 2000});
        } else {
            layer.msg('borrow failed', {icon: 7, time: 2000});
        }
    },
    error: function (data) {
        layer.alert(data.result);
    }
});
```

## 7. 请求参数设计

前端请求参数主要来自三种位置。

| 参数来源 | 示例 | 适用场景 |
| --- | --- | --- |
| 表单序列化 | `$('#addBookForm').serialize()` | 新增图书、新增用户、修改资料 |
| 输入框取值 | `$('#bookId').val()` | 借书、还书 |
| 按钮绑定值 | 删除按钮携带的编号 | 删除分类、删除用户 |

表单序列化适合字段较多的操作，可以减少手动拼接参数的代码。输入框取值适合字段少且需要前端即时校验的操作。按钮绑定值适合列表行操作，例如删除某一条分类或某一个用户。

参数提交时遵循以下原则：

1. 字段名与后端对象属性名保持一致，便于 Spring MVC 自动绑定。
2. 编号类字段在前端先做基本格式校验。
3. 复杂业务校验不放在前端判断，必须由后端业务层完成。
4. 删除类操作必须携带明确的主键编号。

## 8. 响应结果处理

系统中常见的响应结果包括页面、JSON 数组、布尔值和普通操作结果。

| 响应类型 | 使用场景 | 前端处理 |
| --- | --- | --- |
| 页面 | 登录、查询、分页列表 | 浏览器渲染新页面 |
| JSON 数组 | 查询图书分类 | 遍历数组，生成下拉框选项 |
| 布尔值 | 借书、还书、修改资料 | `true` 显示成功，`false` 显示失败 |
| 操作结果 | 新增、删除 | 成功提示后刷新页面，失败提示错误 |

前端不直接访问数据库，也不直接判断复杂业务状态。前端根据后端返回结果决定页面反馈，具体业务规则由业务逻辑层保证。

## 9. 异常处理设计

前端异常处理分为三类。

### 9.1 参数异常

参数异常发生在请求发送前，例如图书编号为空、图书编号不是数字、页码越界等。

处理方式：

1. 在按钮点击事件中读取参数。
2. 使用 `trim()` 去除空白字符。
3. 使用空值判断和 `isNaN()` 判断数字格式。
4. 校验失败时调用 `layer.alert` 或 `layer.msg` 提示。
5. 直接 `return false`，阻止请求继续发送。

这种处理可以减少无效请求，也可以让用户立即看到输入问题。

### 9.2 业务失败

业务失败表示请求已经到达后端，但后端根据业务规则拒绝操作。例如图书不可借、还书记录不存在、删除失败等。

处理方式：

1. 后端返回 `false` 或失败结果。
2. 前端在 `success` 回调中判断返回值。
3. 业务失败不进入 `error` 回调，因为 HTTP 请求本身是成功的。
4. 前端显示失败提示，但不刷新页面。

这种处理可以清楚地区分“请求失败”和“业务不允许”。

### 9.3 请求异常

请求异常表示 HTTP 请求没有正常完成，例如接口不存在、服务器异常、网络中断、返回格式不符合预期等。

处理方式：

1. 在 `$.ajax` 的 `error` 回调中处理。
2. 使用 `layer.alert`、`layer.msg` 或 `alert` 显示错误提示。
3. 不执行成功后的刷新或页面更新逻辑。
4. 调试时通过浏览器 Network 面板查看状态码和响应体。

## 10. 加载状态处理

原有脚本已经在成功和失败回调中提供了用户反馈，但不同脚本的加载状态处理不够统一。为此补充公共 AJAX 辅助脚本：

文件：`scripts/common/ajaxHelper.js`

公共对象：`window.AppAjax`

提供方法：

| 方法 | 作用 |
| --- | --- |
| `post(options)` | 封装 POST AJAX 请求 |
| `loading(message)` | 打开加载提示 |
| `closeLoading(index)` | 关闭加载提示 |
| `fail(message)` | 显示统一失败提示 |

公共封装代码：

```javascript
window.AppAjax = (function () {
    function loading(message) {
        if (window.layer && layer.load) {
            return layer.load(1, {
                shade: [0.12, '#fff'],
                content: message || 'Loading...'
            });
        }
        return null;
    }

    function closeLoading(index) {
        if (index !== null && index !== undefined && window.layer && layer.close) {
            layer.close(index);
        }
    }

    function fail(message) {
        const text = message || 'Request failed. Please try again later.';
        if (window.layer && layer.msg) {
            layer.msg(text, {icon: 2, time: 1800});
        } else {
            alert(text);
        }
    }

    function post(options) {
        let loadingIndex;
        return $.ajax({
            async: options.async === undefined ? true : options.async,
            type: 'post',
            url: options.url,
            dataType: options.dataType || 'json',
            data: options.data || {},
            beforeSend: function () {
                loadingIndex = loading(options.loadingText);
                if (typeof options.beforeSend === 'function') {
                    options.beforeSend();
                }
            },
            success: function (data) {
                if (typeof options.success === 'function') {
                    options.success(data);
                }
            },
            error: function () {
                fail(options.errorText);
                if (typeof options.error === 'function') {
                    options.error();
                }
            },
            complete: function () {
                closeLoading(loadingIndex);
                if (typeof options.complete === 'function') {
                    options.complete();
                }
            }
        });
    }

    return {
        post: post,
        loading: loading,
        closeLoading: closeLoading,
        fail: fail
    };
})();
```

该封装的作用是把加载提示、异常提示和请求生命周期集中管理。后续脚本可以逐步改为使用 `AppAjax.post()`，减少重复编写 `beforeSend`、`error` 和 `complete` 逻辑。

使用示例：

```javascript
AppAjax.post({
    url: '/userBorrowingBook',
    data: {bookId: bookId},
    loadingText: 'Loading...',
    errorText: 'Borrow request failed.',
    success: function (data) {
        if (data.toString() == 'true') {
            layer.msg('borrow success', {icon: 6, time: 2000});
        } else {
            layer.msg('borrow failed', {icon: 7, time: 2000});
        }
    }
});
```

## 11. 前后端接口调用关系

### 11.1 新增图书调用链

```text
管理员填写图书信息
  -> addBook.js 监听提交
  -> POST /addBook
  -> 后端接收 Book 字段
  -> 图书业务逻辑校验分类、库存、编号等数据
  -> 数据访问层写入图书表
  -> 前端提示添加成功或添加失败
```

## 13. 调试检查方法

前后端交互调试应同时观察浏览器和服务端。

浏览器侧检查：

1. 在 Console 面板查看前端脚本是否报错。
2. 在 Network 面板查看请求 URL、请求方法、请求参数和响应内容。
3. 检查请求状态码是否为 `200`。
4. 检查 AJAX 返回值是否与前端判断逻辑一致。
5. 检查页面是否在成功后刷新或更新局部元素。

服务端侧检查：

1. 查看后端是否接收到请求。
2. 查看请求参数是否绑定到对象或方法参数。
3. 查看业务逻辑是否进入预期分支。
4. 查看数据库操作是否执行成功。
5. 查看异常堆栈，判断是参数问题、业务问题还是数据库问题。

## 14. 质量分析

当前交互代码已经覆盖主要业务流程，能够完成基本的前后端数据传递。系统使用 jQuery AJAX 降低了简单交互的实现复杂度，使用 Layui 提供统一风格的弹窗反馈，页面操作对用户来说比较直观。

同时也存在可继续改进的点：

1. 部分 AJAX 请求使用 `async: false`，同步请求会阻塞页面交互，后续应改为异步请求。
2. 不同脚本中的成功和失败提示文本风格不完全统一，后续可统一封装。
3. 业务失败与请求异常已经有区分，但返回结构仍可进一步规范为 `{success, message, data}`。
4. 公共 `AppAjax` 已经提供加载状态和异常处理封装，后续可逐步替换各脚本中的重复 `$.ajax` 代码。
