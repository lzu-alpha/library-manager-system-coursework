# 第 4 项：项目骨架搭建

## 1. 目标

建立清晰、稳定、便于扩展的项目基础骨架，为后续数据访问层、业务逻辑层、接口层和前端页面开发提供统一结构。

项目骨架应满足以下要求：

- 目录结构清晰；
- 各层职责明确；
- 配置文件集中管理；
- 数据库脚本独立存放；
- 静态资源和页面模板分离；
- 基础工具类可复用；
- 便于后续扩展日志、异常处理、分页、数据库连接等通用能力。

## 2. 项目总体结构

项目建议采用以下基础目录结构：

```text
project-root
├── src
│   ├── main
│   │   ├── java
│   │   │   └── com
│   │   │       └── zbw
│   │   │           ├── controller
│   │   │           ├── domain
│   │   │           │   └── Vo
│   │   │           ├── mapper
│   │   │           ├── service
│   │   │           │   └── impl
│   │   │           └── utils
│   │   │               └── page
│   │   └── resources
│   │       ├── db
│   │       ├── mapper
│   │       ├── static
│   │       │   ├── css
│   │       │   ├── images
│   │       │   ├── layui
│   │       │   └── scripts
│   │       ├── templates
│   │       │   ├── admin
│   │       │   ├── common
│   │       │   └── user
│   │       └── application.yml
│   └── test
│       └── java
└── README.md
```

该结构将后端代码、配置文件、SQL 脚本、页面模板、静态资源和测试代码分开存放，便于多人协作和后期维护。

## 3. 后端代码骨架

### 3.1 `controller` 控制层

`controller` 目录用于存放控制器类。

控制层主要职责：

- 接收前端请求；
- 读取请求参数；
- 进行基础参数校验；
- 调用 Service 层完成业务处理；
- 返回页面或 JSON 数据；
- 控制登录状态和访问权限。

命名示例：

```text
UserController
AdminController
BookController
BookCategoryController
BorrowingBooksController
```

示例结构：

```java
@Controller
public class UserController {

    @Resource
    private IUserService userService;

    @PostMapping("/userLogin")
    public String userLogin(String userName, String password, HttpServletRequest request) {
        // 参数校验
        // 调用业务层
        // 返回页面
    }
}
```

控制层不应直接编写 SQL，也不应直接访问 Mapper。

### 3.2 `domain` 实体层

`domain` 目录用于存放实体类，实体类通常与数据库表结构对应。

示例：

```text
User
Book
Department
BorrowingBooks
BookCategory
Admin
```

实体类主要职责：

- 保存数据库表字段；
- 作为 Mapper 查询结果对象；
- 作为 Service 层业务处理的数据载体。

示例：

```java
public class User {
    private Integer userId;
    private String userName;
    private String userPwd;
    private String userEmail;
}
```

实体类中不应编写业务流程代码。

### 3.3 `domain/Vo` 展示对象层

`Vo` 目录用于存放页面展示对象。

VO 对象用于组合多个实体的信息，解决数据库对象与页面展示内容不完全一致的问题。

示例：

```text
BookVo
BorrowingBooksVo
```

使用场景：

- 展示图书时，需要额外显示“是否可借”；
- 展示借阅记录时，需要同时展示用户信息、图书信息、借阅日期和预计归还日期。

示例：

```java
public class BookVo {
    private Integer bookId;
    private String bookName;
    private String bookAuthor;
    private String bookPublish;
    private String isExist;
}
```

### 3.4 `mapper` 数据访问接口层

`mapper` 目录用于存放 MyBatis Mapper 接口。

Mapper 接口主要职责：

- 定义数据库访问方法；
- 与 `resources/mapper` 下的 XML 文件对应；
- 提供增删改查和业务查询接口。

命名示例：

```text
UserMapper
BookMapper
BookCategoryMapper
BorrowingBooksMapper
DepartmentMapper
AdminMapper
```

示例：

```java
public interface BookMapper {
    Book selectByPrimaryKey(Integer bookId);

    int insert(Book record);

    int deleteByPrimaryKey(Integer bookId);

    List<Book> selectBooksByPartInfo(String bookPartInfo);
}
```

Mapper 接口不处理业务规则，只定义数据访问能力。

### 3.5 `service` 业务接口层

`service` 目录用于定义业务接口。

业务接口主要职责：

- 抽象系统业务能力；
- 屏蔽具体实现细节；
- 便于 Controller 调用；
- 便于后续扩展和测试。

命名示例：

```text
IUserService
IBookService
IAdminService
IBookCategoryService
IBorrowingBooksRecordService
```

示例：

```java
public interface IBookService {
    List<BookVo> selectBooksByBookPartInfo(String partInfo);

    Page<BookVo> findBooksByCategoryId(int categoryId, int pageNum);
}
```

### 3.6 `service/impl` 业务实现层

`service/impl` 目录用于存放业务接口的实现类。

业务实现层主要职责：

- 调用 Mapper 完成数据读取和写入；
- 实现登录校验、借书、还书、分页、日期计算等业务规则；
- 处理业务异常；
- 组装页面需要的 VO 数据；
- 控制事务边界。

命名示例：

```text
UserServiceImpl
BookServiceImpl
AdminServiceImpl
BookCategoryServiceImpl
BorrowingBooksRecordServiceImpl
```

示例：

```java
@Service
public class BookServiceImpl implements IBookService {

    @Resource
    private BookMapper bookMapper;

    @Override
    public List<BookVo> selectBooksByBookPartInfo(String partInfo) {
        // 查询图书
        // 判断是否可借
        // 组装 BookVo
    }
}
```

业务实现类不应直接处理页面跳转，页面跳转应由 Controller 负责。

### 3.7 `utils` 工具层

`utils` 目录用于存放通用工具类。

常见工具类包括：

| 工具类型 | 作用 |
|---|---|
| 分页工具 | 保存当前页、每页数量、总页数和数据列表 |
| 日期工具 | 日期格式化、借阅期限计算 |
| 字符串工具 | 空值判断、格式处理 |
| 校验工具 | 参数校验、邮箱格式校验 |
| 响应工具 | 统一接口返回格式 |
| 日志工具 | 统一日志输出 |

分页工具示例：

```java
public class Page<T> {
    private List<T> list;
    private int pageNum;
    private int pageSize;
    private int pageCount;
}
```

工具类应保持通用，不应写入具体业务流程。

## 4. 配置文件骨架

### 4.1 主配置文件

主配置文件为：

```text
src/main/resources/application.yml
```

主要配置内容包括：

- 服务端口；
- 数据库连接；
- MyBatis 映射路径；
- 实体类别名包；
- 页面模板前缀和后缀；
- 静态资源缓存配置。

配置示例：

```yaml
server:
  port: 8080

mybatis:
  mapper-locations: classpath:mapper/*.xml
  type-aliases-package: com.zbw.domain

spring:
  datasource:
    driverClassName: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/library-manager-system?useUnicode=true&serverTimezone=UTC&characterEncoding=utf-8&useSSL=false
    username: root
    password: 123456
```

### 4.2 配置管理要求

配置文件应遵循以下规则：

1. 不在代码中硬编码数据库连接信息；
2. 本地开发、测试和生产环境应使用不同配置；
3. 不提交真实生产密码；
4. 修改配置后应同步更新环境配置文档；
5. 配置项命名应清晰，避免重复配置。

## 5. MyBatis XML 骨架

MyBatis XML 文件统一放在：

```text
src/main/resources/mapper
```

每个 Mapper 接口对应一个 XML 文件。

示例：

| Mapper 接口 | XML 文件 |
|---|---|
| `UserMapper` | `UserMapper.xml` |
| `BookMapper` | `BookMapper.xml` |
| `AdminMapper` | `AdminMapper.xml` |
| `DepartmentMapper` | `DepartmentMapper.xml` |
| `BookCategoryMapper` | `BookCategoryMapper.xml` |
| `BorrowingBooksMapper` | `BorrowingBooksMapper.xml` |

XML 文件应包含：

- `namespace`；
- `resultMap`；
- 通用字段列表；
- 新增 SQL；
- 删除 SQL；
- 修改 SQL；
- 主键查询 SQL；
- 条件查询 SQL；
- 分页查询 SQL。

基本结构示例：

```xml
<mapper namespace="com.zbw.mapper.BookMapper">

    <resultMap id="BaseResultMap" type="com.zbw.domain.Book">
        <id column="book_id" property="bookId"/>
        <result column="book_name" property="bookName"/>
    </resultMap>

    <sql id="Base_Column_List">
        book_id, book_name, book_author, book_publish
    </sql>

    <select id="selectByPrimaryKey" resultMap="BaseResultMap">
        select
        <include refid="Base_Column_List"/>
        from book
        where book_id = #{bookId}
    </select>

</mapper>
```

## 6. 数据库脚本骨架

数据库脚本统一放在：

```text
src/main/resources/db
```

数据库脚本应包含：

- 建库语句；
- 建表语句；
- 外键约束；
- 初始化数据；
- 必要索引；
- 脚本执行说明。

推荐拆分方式：

```text
db
├── schema.sql
├── data.sql
└── README.md
```

如果项目较小，也可以使用一个完整 SQL 文件保存建表和初始化数据。

数据库脚本编写要求：

1. 表名和字段名使用小写字母加下划线；
2. 主键字段明确；
3. 外键关系清晰；
4. 初始化数据可以重复导入或有明确执行说明；
5. 字符集统一使用 UTF-8；
6. 涉及删除表时要注意外键约束顺序。

## 7. 前端页面骨架

### 7.1 页面模板目录

页面模板统一放在：

```text
src/main/resources/templates
```

推荐结构：

```text
templates
├── index.html
├── admin
│   ├── index.html
│   ├── addBook.html
│   ├── addCategory.html
│   ├── addUser.html
│   ├── showBooks.html
│   └── showUsers.html
├── user
│   ├── index.html
│   ├── findBook.html

│   ├── returnBooks.html
│   └── userMessage.html
└── common
    ├── admin_header.html
    ├── user_header.html
    └── footer.html
```

页面分类说明：

| 目录 | 作用 |
|---|---|
| `templates/admin` | 管理员功能页面 |
| `templates/user` | 普通用户功能页面 |

### 7.2 静态资源目录

静态资源统一放在：

```text
src/main/resources/static
```

推荐结构：

```text
static
├── css
├── images
├── layui
└── scripts
    ├── admin
    └── user
```

静态资源分类说明：

| 目录 | 作用 |
|---|---|
| `css` | 页面样式文件 |
| `images` | 图片资源 |
| `layui` | Layui 前端组件库 |
| `scripts/admin` | 管理员页面脚本 |
| `scripts/user` | 用户页面脚本 |

页面中不应堆积大量 JavaScript 代码，页面交互逻辑应放入 `static/scripts` 下的独立 JS 文件。

## 8. 测试代码骨架

测试代码统一放在：

```text
src/test/java
```

测试代码应覆盖：

- Mapper 数据访问；
- Service 业务逻辑；
- Controller 请求处理；
- 数据库连接；
- 分页查询；
- 借书、还书等核心业务流程。

测试类命名示例：

```text
UserTest
BookServiceTest
BookCategoryMapperTest
BorrowingBooksMapperTest
DemoApplicationTests
```

测试代码要求：

1. 测试方法命名清晰；
2. 使用断言判断结果；
3. 测试数据可重复执行；
4. 不依赖不稳定的本地临时数据；
5. 核心业务必须有正向和异常场景测试。

## 9. 基础工具类设计

### 9.1 分页工具类

分页是后台管理和图书查询中的通用功能，建议定义统一分页对象。

示例：

```java
public class Page<T> {
    private List<T> list;
    private int pageNum;
    private int pageSize;
    private int pageCount;
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `list` | 当前页数据列表 |
| `pageNum` | 当前页码 |
| `pageSize` | 每页记录数 |
| `pageCount` | 总页数 |

分页对象可用于图书列表、用户列表、图书分类列表和借阅记录列表。

### 9.2 日志工具

项目应统一使用日志框架输出运行日志。

日志级别建议：

| 级别 | 使用场景 |
|---|---|
| `debug` | 开发调试信息 |
| `info` | 正常业务流程信息 |
| `warn` | 可恢复的异常或风险 |
| `error` | 系统异常或业务失败 |

示例：

```java
log.info("用户登录成功，userId={}", user.getUserId());
log.error("借书失败，bookId={}", bookId, e);
```

不应使用大量 `System.out.println` 作为正式日志。

### 9.3 异常处理工具

项目应建立统一异常处理机制。

建议包括：

- 业务异常类；
- 统一错误码；
- 全局异常处理器；
- 统一响应对象。

示例：

```java
public class BusinessException extends RuntimeException {
    public BusinessException(String message) {
        super(message);
    }
}
```

Controller 层可通过全局异常处理器统一返回错误信息，避免页面或接口直接暴露系统异常。

### 9.4 统一响应对象

接口返回 JSON 数据时，建议使用统一响应结构。

示例：

```java
public class Result<T> {
    private boolean success;
    private String message;
    private T data;
}
```

示例返回：

```json
{
  "success": true,
  "message": "操作成功",
  "data": {}
}
```

统一响应结构可以降低前后端联调成本。

## 10. 项目骨架搭建流程

项目骨架搭建可按以下步骤进行：

1. 创建 Java Web 工程；
2. 建立 `controller`、`domain`、`mapper`、`service`、`service/impl`、`utils` 包；
3. 创建 `resources/mapper` 存放 MyBatis XML；
4. 创建 `resources/db` 存放数据库脚本；
5. 创建 `resources/templates` 存放页面模板；
6. 创建 `resources/static` 存放静态资源；
7. 编写 `application.yml` 配置端口、数据库连接和 MyBatis；
8. 编写分页、日志、异常处理等基础工具类；
9. 编写基础测试类验证项目能正常加载；
10. 提交项目骨架代码到远程仓库。

## 11. 骨架完整性检查

项目骨架完成后，应进行以下检查：

| 检查项 | 要求 |
|---|---|
| 目录结构 | 后端、前端、配置、数据库、测试目录清晰 |
| 配置文件 | `application.yml` 能正确加载 |
| 数据库脚本 | SQL 文件位置明确，可执行 |
| Mapper XML | XML 文件与 Mapper 接口对应 |
| 页面模板 | 页面按角色和功能分类 |
| 静态资源 | CSS、JS、图片分类存放 |
| 工具类 | 分页、日志、异常等基础能力可复用 |
| 测试代码 | 至少能验证项目上下文加载 |
| 编码 | 所有文件统一 UTF-8 |
| 版本管理 | 无编译产物和临时文件误提交 |
